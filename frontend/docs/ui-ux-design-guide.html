1. Platform Overview

1.1 What is CosmicForge RTC?

CosmicForge RTC is an enterprise-grade, self-hosted real-time video communication platform. It enables organizations and third-party applications to embed secure video conferencing capabilities into their workflows.

Key Value Propositions
  • Self-Hosted: Full data sovereignty and privacy control
  • API-First: Seamless third-party integration via API keys
  • Enterprise Features: Waiting rooms, host controls, audit logs
  • Modern Stack: WebRTC-based with LiveKit SFU architecture

1.2 Platform Architecture

The platform follows a control plane + data plane separation:

Layer Technology Purpose
Frontend Next.js + TypeScript User interface, meeting room UI
Control Plane Rust (Axum) API, authentication, state management
Data Plane LiveKit (WebRTC SFU) Audio/video streaming
Database PostgreSQL Persistent storage

1.3 Design Goals

Simplicity

Clean, intuitive interface that requires minimal learning curve

Professional

Enterprise-ready appearance suitable for business contexts

Accessible

Responsive

Seamless experience across desktop, tablet, and mobile

2. Target Users & Personas

2.1 Primary Personas

Persona 1: Meeting Host (Sarah)

Role: Team Lead / Manager

Goals: Schedule and manage team meetings efficiently, control participant access, ensure productive meetings

Pain Points: Complex scheduling, poor participant management, lack of control during meetings

Key Features Used: Meeting creation, waiting room, participant controls, screen sharing

Persona 2: Meeting Participant (Alex)

Role: Team Member / Client

Goals: Join meetings quickly, communicate clearly, share content when needed

Pain Points: Complicated join process, audio/video issues, confusing controls

Key Features Used: Quick join, media controls, chat, reactions

Persona 3: Developer / Integrator (Marcus)

Role: Software Developer

Goals: Integrate video calls into existing applications, manage API access

Pain Points: Poor documentation, complex authentication, limited customization

Key Features Used: API key management, meeting creation via API, embedded join links

3.6 Meeting States

SCHEDULED → ONGOING → ENDED

Transitions:

  • Scheduled → Ongoing: First participant joins after start time
  • Ongoing → Ended: Host ends meeting or auto-end at scheduled end time
  • Scheduled → Cancelled: Host deletes meeting before start

4. User Flows

4.1 Registration Flow

Landing Page → Sign Up Form → Verification Email → Enter Code → Dashboard
Design Note

Verification uses a 6-digit code, not a link. Design an input field with 6 separate boxes for each digit (similar to 2FA inputs).

4.2 Login Flow

Login Page → Email + Password → Dashboard

Alternative: Google OAuth (single button, redirects to Google)

4.3 Password Reset Flow

Forgot Password → Enter Email → Enter 6-Digit Code → New Password → Login

4.4 Create Meeting Flow

Dashboard → New Meeting → Fill Details → Copy Link → Share

Meeting Form Fields:

  • Title: Text input (required)
  • Start Time: Date/time picker (must be at least 1 min in future)
  • End Time: Date/time picker (max 24h after start)
  • Privacy: Toggle (Public / Private with waiting room)

4.5 Join Meeting Flow (Public)

Join Page → Enter Code/Link → Preview Screen → Join Meeting

Preview Screen Elements:

  • Camera preview (self-view)
  • Display name input (editable)
  • Microphone selection dropdown
  • Camera selection dropdown
  • Audio test button
  • "Join Now" button

4.6 Join Meeting Flow (Private - Waiting Room)

Join Page → Preview Screen → Waiting Room → Admitted → Meeting Room
Waiting Room UX

Display a clean "Please wait" message with meeting title. Show admission status. Allow participants to leave while waiting. Consider showing their position or "Host will let you in shortly."

4.7 In-Meeting Flow (Host)

Meeting Room → Manage Waiting Room → Control Participants → End Meeting

Host-Only UI Elements:

  • Waiting room notification badge
  • Admit/Deny buttons for each waiting participant
  • Kick option in participant context menu
  • "End Meeting for All" button
  • Mute participant option

4.8 API Key Management Flow

Dashboard → Developer Settings → Create Key → Copy Key → Use in App
Security Note

API keys should only be shown once at creation. After that, only show masked version (last 4 chars). Include a "Copy" button and warning about secure storage.

7.6 Animation Guidelines

  • Duration: 150-300ms for micro-interactions, 300-500ms for page transitions
  • Easing: ease-out for entering elements, ease-in for exiting
  • Motion: Subtle scale and fade for modals, slide for side panels
  • Reduce Motion: Respect prefers-reduced-motion media query

8.2 Layout Adaptations

Dashboard

  • Desktop: Sidebar visible, content area with multi-column grid
  • Tablet: Collapsible sidebar, 2-column grid
  • Mobile: Bottom navigation or hamburger menu, single column

Meeting Room

  • Desktop: Video grid with side panel (chat/participants), bottom control bar
  • Tablet: Smaller video tiles, slide-out panels
  • Mobile: Stack layout, full-screen speaker view, swipe for panels, floating controls

Forms

  • Desktop: Inline labels, multi-column where appropriate
  • Mobile: Stacked labels, full-width inputs, larger touch targets (44px min)

8.3 Mobile-Specific Considerations

Touch Targets

Minimum 44x44px for all interactive elements. Add padding if needed.

Thumb Zones

Primary actions in bottom half of screen. Use bottom sheets instead of dropdowns.

Orientation

Support both portrait and landscape. Optimize meeting room for landscape.

9. Accessibility Requirements

9.1 WCAG 2.1 AA Compliance

Color Contrast

  • Normal text: minimum 4.5:1 contrast ratio
  • Large text (18px+ bold or 24px+): minimum 3:1
  • UI components and graphics: minimum 3:1
  • Never use color alone to convey information

Keyboard Navigation

  • All interactive elements must be keyboard accessible
  • Visible focus indicators on all focusable elements
  • Logical tab order following visual layout
  • Escape key to close modals and menus
  • Keyboard shortcuts for common meeting actions

Screen Reader Support

  • Semantic HTML elements (button, nav, main, etc.)
  • ARIA labels for icon-only buttons
  • Live regions for dynamic content (new chat messages, participant join/leave)
  • Alt text for meaningful images
  • Proper heading hierarchy (h1 > h2 > h3)

9.2 Meeting Room Accessibility

10. SaaS Best Practices

10.1 Onboarding Experience

  • Progressive Disclosure: Don't overwhelm new users. Show essential features first.
  • Empty States: Provide helpful guidance when no data exists (e.g., "No meetings yet. Create your first meeting!")
  • Tooltips: Contextual help for advanced features
  • Product Tour: Optional guided tour for first-time users

10.2 Trust & Security Signals

  • Display "End-to-end encrypted" badge in meeting room
  • Show privacy indicators for private meetings
  • Clear data handling messaging during signup
  • API key security warnings and best practices

10.3 Error Handling

  • Friendly Messages: Avoid technical jargon. "Something went wrong" with clear next steps.
  • Recovery Options: Always provide a way forward (retry, go back, contact support)
  • Inline Validation: Real-time form validation with helpful hints
  • Connection Issues: Clear indicators for network problems during meetings

10.4 Performance Perception

  • Skeleton Loaders: Show content structure while loading
  • Optimistic Updates: Update UI immediately, sync in background
  • Progress Indicators: Show progress for longer operations
  • Instant Feedback: Button states change immediately on click

10.5 Engagement Patterns

  • Quick Actions: "New Meeting" button always accessible
  • Copy to Clipboard: One-click copy for meeting links and codes
  • Recent Items: Quick access to recent meetings
  • Notifications: In-app notifications for meeting reminders, waiting room

10.6 Consistency Checklist

Area Guideline
Language Consistent terminology: "Meeting" not "Call/Room/Session" mixed
Actions Primary action always on the right, cancel on left
Icons Same icon for same action throughout (mic icon always = mute)
Colors Red = destructive, Green = success, Primary = main action
Feedback Toast for temporary, modal for important/blocking

10.7 Key Metrics to Support

Design should support measuring:

  • Time to create first meeting (onboarding success)
  • Meeting join success rate (friction points)
  • Feature adoption (screen share, chat usage)
  • User retention (return visit rate)

10.8 Final Recommendations

Prioritize the Core Experience

The meeting room is the heart of the product. Invest heavily in making video/audio quality perception excellent, controls intuitive, and the experience distraction-free.

Reduce Cognitive Load

Every additional option is a decision. Keep the default path simple. Advanced features should be discoverable but not in the way.

Design for Reliability

Users must trust the platform. Clear status indicators, graceful error handling, and consistent behavior build trust over time.
